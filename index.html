A metadata schema connecting structured product attributes to user skin profiles and usage contexts — designed to support recommendation, not just cataloging.

Author  Seo Jayoung ID  2022312720 Namespace  skc: Elements  15 Standards  DC · Schema.org · FOAF · GoodRelations

Section A

Opening Narrative

Metadata in the Beauty Domain

The beauty and cosmetics industry has historically relied on product catalogs organized around brand hierarchies and ingredient lists — structures inherited from pharmaceutical and retail databases. As e-commerce expanded through the 2000s, platforms began applying faceted search to skincare, exposing for the first time how poorly existing data structures handled the complexity of skin-type matching. A toner suited for dry skin and one suited for oily skin might share identical category tags while being fundamentally incompatible substitutes for any given user.

The global rise of K-beauty through the 2010s accelerated this problem. Korean skincare introduced multi-step routines, ingredient-focused marketing, and a level of product specificity — skin concerns, functional ingredient naming, routine step positioning — that generic retail metadata was not equipped to represent. Platforms began improvising: some added free-text skin concern filters, others used tag clouds, others left the matching work entirely to user reviews. None of these approaches produced structured, interoperable data.

This schema is a response to that gap. It draws on library and information science metadata design — precision of definition, controlled vocabularies, explicit cardinality — and applies it to the domain of Korean skincare product description.

Intended Users

The primary users are developers of skincare recommendation platforms, data analysts, and product managers in the beauty and tech industries who need structured data to build or evaluate personalized recommendation systems. Secondary users include beauty brands looking to represent their product lines in a machine-readable format, and researchers in consumer behavior or information science studying data-driven service design.

The schema operates one layer below the user interface: it is the data structure that makes a well-designed recommendation UI possible.

Development Notes

The schema was developed across four drafts. Draft 1 established the domain scope and user needs. Draft 2 identified ten core objects in the test corpus and surveyed existing metadata standards. Draft 3 produced the full data dictionary with controlled vocabularies for all fifteen elements. Draft 4 translated the dictionary into a machine-readable XSD file and produced three validating XML instance records.

A key design tension throughout was between expressiveness and validation strictness. Controlled vocabularies enforce consistency and enable automated matching, but require every valid value to be anticipated at schema-design time. The chosen approach — strict enumeration for classification elements, free-text for descriptive elements — reflects a deliberate trade-off: recommendation-critical fields (skin type, concern, routine step) use controlled vocabularies; human-readable description fields remain open.

Theoretical Considerations

This schema operates on a user-product matching model rather than a simple cataloging model. The targetSkinType and targetSkinConcern elements are not purely descriptive — they encode a compatibility assertion between product and user. The usageContext element extends this further into situational matching, capturing environmental conditions (season, climate, time of day) that affect product suitability. This draws on context-aware recommender systems research, where situational factors have been shown to significantly affect user satisfaction with recommendations.

Section B

Relevant Metadata Standards

Four existing standards are implemented alongside the locally defined skc: namespace for domain-specific elements.

dc:

Dublin Core

Used for dc:title and dc:description. The only external standard that can be directly imported via xs:import and referenced with ref=, as its XSD publishes globally declared elements. Chosen for its universality and broad interoperability.

schema:

Schema.org

Source vocabulary for the brand element (schema:brand). The dominant standard for structured product data on the web. As Schema.org does not publish a standard XSD with globally declared elements, it is re-declared locally as skc:brand with the source URI annotated.

foaf:

FOAF

The concept of foaf:age is adapted for the target age range element, applied at the product level to describe the user demographic a product is suited for. As FOAF does not publish an XSD file, it is locally re-declared as skc:targetAgeRange.

gr:

GoodRelations

The pricing element is modeled on gr:hasPriceSpecification, including GoodRelations' sub-element naming convention (hasCurrencyValue, hasCurrency). A well-established e-commerce ontology whose pricing structure maps directly to this schema's needs.

Data Dictionary

dc:titleProduct Title
URI
http://purl.org/dc/elements/1.1/title
Definition
The official name of the skincare product as printed on the label or listed on the manufacturer's page. Recorded in the original language and script.
Data values
Free-text string. If both Korean and English names exist, the primary name goes here; secondary via dc:alternative.
Vocab
None
Cardinality
Mandatory | Not repeatable
skc:brandBrand
URI
https://example.org/skincare#brand  ·  source: https://schema.org/brand
Definition
The brand or manufacturer behind the product. Used to group products by brand and support brand-level filtering.
Vocab
Round Lab · COSRX · Anua · Beauty of Joseon · Innisfree · HERA · Some By Mi · Klairs · Laneige · Sulwhasoo
Cardinality
Mandatory | Not repeatable
Comments
For co-branded products, record the primary brand here and note the secondary in dc:description.
skc:productCategoryProduct Category
URI
https://example.org/skincare#productCategory
Definition
The product's functional category within a skincare routine, based on Korean skincare step classification.
Vocab
Cleanser · Toner / Softener · Essence · Serum · Ampoule · Lotion (Emulsion) · Cream · Eye Cream · Sunscreen · Mask Sheet · Sleeping Pack · Exfoliator · Mist · Primer · Cushion Foundation · BB Cream
Cardinality
Mandatory | Not repeatable
Comments
Some products straddle two categories (e.g., toner-essence hybrids). In that case, record the primary category here and note the secondary in dc:description. Closely tied to skc:routineStep.
skc:targetSkinTypeTarget Skin Type
URI
https://example.org/skincare#targetSkinType
Definition
The skin type(s) the product is suited for. One of the primary matching criteria in recommendation.
Vocab
Dry · Oily · Combination · Sensitive · Normal · Acne-prone · Mature · All skin types
Cardinality
Mandatory | Repeatable
skc:targetSkinConcernTarget Skin Concern
URI
https://example.org/skincare#targetSkinConcern
Definition
Skin concerns the product addresses. Together with targetSkinType, forms the core of the product-user matching logic.
Vocab
Hydration · Brightening · Anti-aging · Pore minimizing · Sebum control · Acne / Blemish · Soothing / Redness relief · Barrier repair · Hyperpigmentation · Sun protection · Firming · Dark circles
Cardinality
Mandatory | Repeatable
skc:keyIngredientKey Ingredient
URI
https://example.org/skincare#keyIngredient
Definition
A notable active ingredient recorded by INCI name with a functional role. Sub-elements: inci, role, concentration (optional %).
Vocab (role)
Humectant · Emollient · Occlusive · Antioxidant · Exfoliant (AHA/BHA) · Anti-inflammatory · Brightening agent · UV filter · Antimicrobial · Film-forming · Preservative · Fragrance
Cardinality
Optional | Repeatable
Comments
Sub-elements: skc:keyIngredient/inci (INCI name), skc:keyIngredient/role (from Appendix D), skc:keyIngredient/concentration (numeric %, optional). Record the top 3–5 active ingredients; full INCI lists are not required here.
skc:avoidIngredientFlagged Ingredient
URI
https://example.org/skincare#avoidIngredient
Definition
An ingredient present in the product that may be problematic for certain users. Used for negative filtering in recommendation.
Vocab
Fragrance / Parfum · Alcohol denat. · Parabens · Sulfates (SLS/SLES) · Silicone · Essential oils · Retinol · Hydroquinone · Oxybenzone · Formaldehyde releasers · Mineral oil · Artificial colorants
Cardinality
Optional | Repeatable
Comments
If the product is explicitly labeled "fragrance-free" or "paraben-free," leave the corresponding value empty rather than omitting the element entirely. Works alongside a user profile's avoidance preferences for matching purposes.
skc:textureTypeTexture Type
URI
https://example.org/skincare#textureType
Definition
The sensory texture of the product formula. A significant preference factor — e.g., oily skin users often prefer watery or gel textures.
Vocab
Watery · Gel · Lightweight lotion · Rich lotion · Cream · Balm · Oil · Serum (thin) · Serum (viscous) · Foam · Cushion · Stick · Mist
Cardinality
Optional | Not repeatable
Comments
Values should come from manufacturer descriptions or consistent review data, not individual experience. For two-phase products (e.g., a shake-to-mix mist), record the dominant texture upon application.
skc:functionalClaimFunctional / Regulatory Claim
URI
https://example.org/skincare#functionalClaim
Definition
A formal product claim — regulatory (SPF/PA, 기능성화장품) or certification-based (vegan, cruelty-free). Sub-elements: claimType, claimValue.
Vocab
SPF · PA · 기능성화장품 · Dermatologist-tested · Hypoallergenic · Vegan · Cruelty-free · EWG verified · Fragrance-free · Non-comedogenic
Cardinality
Optional | Repeatable
Comments
For sunscreen records, SPF and PA values should be treated as effectively mandatory. Sub-elements: skc:functionalClaim/claimType, skc:functionalClaim/claimValue.
skc:routineStepRoutine Step
URI
https://example.org/skincare#routineStep
Definition
The step in a standard Korean 10-step skincare routine where this product is used.
Vocab
Step 1: Oil cleanser · Step 2: Water-based cleanser · Step 3: Exfoliator · Step 4: Toner · Step 5: Essence · Step 6: Serum / Ampoule · Step 7: Sheet mask · Step 8: Eye cream · Step 9: Moisturizer · Step 10: Sunscreen (AM) / Sleeping pack (PM)
Cardinality
Mandatory | Not repeatable
Comments
Products used in both AM and PM routines should note this in dc:description. Linked to skc:compatibleProduct for routine sequencing logic.
skc:usageContextUsage Context
URI
https://example.org/skincare#usageContext
Definition
Environmental or situational conditions the product is suited for. Expressed as a dimension–value pair via the @dimension attribute and element text content.
Dimensions
Season · Climate · Time of day · Setting · Special condition
Values
Spring · Summer · Fall · Winter · Year-round · Humid · Dry · Cold · Hot · Temperate · AM routine · PM routine · Both · Indoor · Outdoor · Travel · Post-procedure · Pre-event · Pregnancy-safe
Cardinality
Optional | Repeatable
skc:targetAgeRangeTarget User Age Range
URI
https://example.org/skincare#targetAgeRange  ·  source: http://xmlns.com/foaf/0.1/age
Definition
The age bracket of users the product is most suited for. Applied at the product level — not referring to an actual person's age.
Vocab
Teens (13-19) · 20s · 30s · 40s+ · All ages
Cardinality
Optional | Repeatable
Comments
When matched against a FOAF user profile, the product's age range value is compared against the user's age bracket for compatibility. Anti-aging products will typically target 30s–40s+.
skc:priceSpecificationPrice Specification
URI
https://example.org/skincare#priceSpecification  ·  source: gr:hasPriceSpecification
Definition
The retail price of the product. Sub-elements follow GoodRelations naming: hasCurrencyValue (xs:decimal) and hasCurrency (ISO 4217).
Vocab
Currency: KRW · USD · EUR
Cardinality
Optional | Not repeatable
Comments
Sub-elements follow the GoodRelations spec: gr:hasCurrencyValue (numeric) and gr:hasCurrency (ISO 4217 string). Price data changes often, so stamping with dc:date when recorded is recommended.
dc:descriptionDescription
URI
http://purl.org/dc/elements/1.1/description
Definition
A short free-text description covering purpose, texture, and anything not captured by structured elements. Ideally 1–3 sentences.
Vocab
None
Cardinality
Optional | Not repeatable
skc:compatibleProductCompatible Product
URI
https://example.org/skincare#compatibleProduct
Definition
A reference to another product recommended for use in the same routine. Sub-elements: productRef (xs:IDREF) and position (before / after).
Vocab
position: before · after
Cardinality
Optional | Repeatable
Comments
Sub-elements: skc:compatibleProduct/productRef (URI/ID), skc:compatibleProduct/position (before / after). The routine set object in the corpus is essentially a curated list of these references in step order.

Sample XML Records

Three sample records from the test corpus, all validating against the hosted XSD. Records are linked via xs:IDREF through the compatibleProduct element.

Record 1 — Round Lab Dokdo Toner

product001 (excerpt)
<skincareProduct productID="product001">
  <dc:title>Round Lab Dokdo Toner</dc:title>
  <brand>Round Lab</brand>
  <productCategory>Toner / Softener</productCategory>
  <targetSkinTypes>
    <targetSkinType>Dry</targetSkinType>
    <targetSkinType>Sensitive</targetSkinType>
    <targetSkinType>Normal</targetSkinType>
  </targetSkinTypes>
  <targetSkinConcerns>
    <targetSkinConcern>Hydration</targetSkinConcern>
    <targetSkinConcern>Barrier repair</targetSkinConcern>
    <targetSkinConcern>Soothing / Redness relief</targetSkinConcern>
  </targetSkinConcerns>
  <keyIngredients>
    <keyIngredient>
      <inci>Dokdo Sea Water</inci>
      <role>Humectant</role>
    </keyIngredient>
    <keyIngredient>
      <inci>Sodium Hyaluronate</inci>
      <role>Humectant</role>
    </keyIngredient>
  </keyIngredients>
  <avoidIngredients/>
  <textureType>Watery</textureType>
  <functionalClaims>
    <functionalClaim>
      <claimType>Fragrance-free</claimType>
      <claimValue>Fragrance-free</claimValue>
    </functionalClaim>
    <functionalClaim>
      <claimType>EWG verified</claimType>
      <claimValue>EWG verified</claimValue>
    </functionalClaim>
  </functionalClaims>
  <routineStep>Step 4: Toner</routineStep>
  <usageContexts>
    <usageContext dimension="Season">Year-round</usageContext>
    <usageContext dimension="Time of day">Both</usageContext>
  </usageContexts>
  <priceSpecification>
    <hasCurrencyValue>18000</hasCurrencyValue>
    <hasCurrency>KRW</hasCurrency>
  </priceSpecification>
  <compatibleProducts>
    <compatibleProduct>
      <productRef>product002</productRef>
      <position>after</position>
    </compatibleProduct>
  </compatibleProducts>
</skincareProduct>
